# Upgrade Durability Audit — 2026-07-16

Point-in-time audit of what every service's state surfaces are and what an
image bump preserves versus destroys. Produced by a mapped-then-adversarially-
verified multi-agent review of the working tree at commit `f6e6210` (each
mapper claim was independently challenged; corrected facts appear here, and
claims that rest on upstream behavior unprovable from this source tree are
marked **[REHEARSE]**). File:line citations reference that commit and will
drift. The Blue/Green implications in §3 are the state-map input the
[scaling and HA posture](../high-availability.md) planning exercise calls for.


Scope: every long-running service plus the one-shot initializer, across six clusters (postgres-keycloak, vault-plane, inference-plane, chat-plane, edge-plane, observability-plane, services-and-init). Each mapper claim was adversarially verified; where the verifier corrected a claim, the corrected fact appears below. Compose line numbers use the verifier-corrected values for the current working tree (mapper citations were systematically ~9–17 lines stale).

## 2. Ranked upgrade risks (image-bump breakage of a live deployment)

Refuted claims are excluded. Items marked **[REHEARSE]** rest on upstream behavior unprovable from this source tree and need a live rehearsal before being relied on.

1. **Botched Postgres major bump → silent fresh cluster → realm reimport → Anthropic WIF trust severed.** The "fail-closed by construction" claim was refuted: the current pinned image declares no Dockerfile `VOLUME` (verified live), but a future DHI major that does gets a writable anonymous volume at its new versioned PGDATA despite `read_only:true` — initdb runs, `01-init-databases.sh` creates empty DBs, Keycloak imports templates into the empty DB, and the stack comes up green with users/keys gone **and** the Console-registered WIF JWKS invalidated (`docs/anthropic-wif-bootstrap.md:71-75`). The only construction-grade guards are the validator pin (`scripts/validate-compose.sh:1489-1490` — gating every bump, minor included) and the ≤24 h backup gate (`main.yml:2163-2170`, `scripts/pre-upgrade-check.sh`). **[REHEARSE]** — future DHI PGDATA/VOLUME/entrypoint conventions are unverifiable from source; any major bump must be rehearsed on a disposable host with `docker image inspect` of the new digest first.

2. **One-way schema migrations with no mechanical rollback, and no version guard on restore.** Keycloak (Liquibase), LiteLLM (Prisma), Open WebUI (SQLite), and Grafana all forward-migrate on first start of a bumped image; image rollback cannot undo it (`docs/operations.md:1049-1056`, `docs/test-runbook.md:942-944`, compose:564-566). `scripts/state-restore.sh` performs **no version-compatibility check** against the receipt's recorded `postgres_version` (verified: zero hits), and the converge's own rollback manifest preserves the old image ID — making the unsafe revert one compose edit away (`main.yml:2806-2851`). **[REHEARSE]** — behavior of each app against a newer-than-binary schema, Prisma's migration mode, and restore-into-newer-major are all upstream; `docs/operations.md:1050-1057` already mandates the rehearsal that no automated test performs.

3. **Every litellm recreate hard-downs inference behind a green healthcheck until key-rotator re-mints.** The in-memory `credential_list` is never rehydrated from surviving DB rows; `/health/readiness` gates only on DB reachability (compose:539-553). Recovery depends entirely on a live single-replica key-rotator with the vendor row enabled (`scheduler.py:254-265,284-323`; disabled rows skipped at :314-316) — ~15-20 s when healthy, **indefinite** if the rotator is down or the vendor paused. The self-heal is pinned to v1.91.3 `/credentials` quirks (`litellm_client.py:123-244`); **[REHEARSE]** any litellm bump must re-verify those semantics live before trust.

4. **Vault image bump = sealed window, and the rotator's sealed-start retry proof is still PENDING.** Vault always restarts sealed; key-rotator is already running (`depends_on vault: service_started`, compose:996) and the previously deployed version consumed two sealed-window jobs as failures. The remediation has passed only its available-Vault path — the live sealed→unsealed proof is explicitly pending (`docs/operations.md:237-244,359-366`). **[REHEARSE]** — this is the one gap the repo itself flags as unproven.

5. **Key-rotator restart near Anthropic WIF token expiry lapses inference auth.** The 80%-of-`expires_in` refresh lives only in APScheduler's in-memory jobstore; on restart the rebuilt IntervalTrigger first fires a full interval (default 3000 s, `db.py:59`), and presence-reconcile only acts when the credential name is *absent* from LiteLLM — not when a loaded token is near expiry (`scheduler.py:783-792,308-323`). The 90%-lifetime alert detects but does not prevent. Confirmed end-to-end in code.

6. **Traefik pin divergence is completely unguarded, with a shared-tag blast radius.** No contract test or `validate-compose.sh` assertion pins the traefik BASE_IMAGE/PATCH_IMAGE digests, the Dockerfile ARG defaults, or their equality (grep-verified zero hits — the convention exists for other images but not traefik). The two services carry duplicate copies of identical build args (compose:130-136 vs 181-186) feeding one shared tag; a partial bump silently ships whichever build ran last, and one bad build recreates **both** the internal and admin edges in the same converge.

7. **Envoy trust store is image-baked; an interrupted converge leaves stale CA bundles live with no warning.** Bundles and `envoy.yaml` are `COPY`'d at build (Dockerfile:14-15) — no bind mount, no digest label — so the planner-triggered rebuild is the only activation path; a converge that syncs new `.pem` files but dies before the build task (`main.yml:451-474` vs `:2853-2868`) keeps serving the old trust store. Additionally: `ENVOY_CONFIG` env redirection bypasses the argv-only override rejection (`entrypoint.go:50-53,79-87`), and **[REHEARSE]** whether `service_healthy` ordering holds under the converge's `--no-deps` is Compose-runtime behavior not provable from the repo.

8. **Grafana's exact-inventory verify fails the bump converge itself if any operator ever clicked "New dashboard".** Verify requires exactly 9 unified-storage rows with no folder filter, plus name-allowlist and `managedBy` annotation checks (`verify/tasks/main.yml:1642-1724`) — and every auth-proxy user is auto-provisioned org **Admin** (compose:1473), so UI-created dashboards are both possible and persistent in `grafana_data`. Sweep for unmanaged dashboards before any observability bump.

9. **Realm-template drift and OIDC-secret rotation asymmetry survive every bump.** `--import-realm` skips existing realms; post-ceremony, redirect-URI reconciliation degrades to a warning (`main.yml:3788-3805`) and nothing compares the live realm to `aigw-realm.json` except the single aigw-chat gate (`verify/tasks/main.yml:1879-1949`). Rotating any `*_OIDC_CLIENT_SECRET` in the overlay updates only the RP env — the Keycloak-side secret from import time stays until the identity ceremony reruns (`identity.py:806-826`; `docs/identity-operations.md:258-263,418-421`). Logins break; no image bump ever fixes this.

10. **A one-line edit to the shared health-probe context recreates ~14 services simultaneously, including Vault (which reseals).** Documented in-role (`main.yml:2705-2711`). Compounding: final DHI image builds skip on PRs without dhi.io credentials — a green PR does not prove a bumped base builds; failure surfaces at converge time.

11. **Lab rehearsals exercise strictly less protection than production.** The backup gate is off (`ansible/inventory/lab.yml:16` vs default-true `group_vars/all.yml:376`), and the lab overlay weakens keycloak's postgres dependency to `service_started` (lab overlay :142-144 vs `service_healthy` at compose:770). An upgrade that passes in the lab has not exercised the production gate or startup ordering. For the lab specifically, a samba-ad bump has **no ldb compatibility gate** and, via `service_healthy, required: true` (lab.yml:142-147), a failed one blocks Keycloak's recreate. **[REHEARSE]** in-place sam.ldb upgrades.

12. **Keycloak user-session survival across a recreate — PROVEN (2026-07-16).** Rehearsed: five active `lab-developer` online sessions, force-recreate the Keycloak container (new container ID; only the Postgres-backed realm DB persists), and all five sessions return with identical IDs and start times. Keycloak 26's default persistent-user-sessions writes online sessions to the DB, so an image bump (which likewise only recreates the container) preserves logged-in users. The residual risk narrows to a one-way schema migration corrupting the session tables — covered by risk 2's rehearsal mandate, not a separate session concern.

13. **Loki/Tempo storage-format and shutdown hazards.** Loki: `schema_config` is append-only (mutating the v13 entry corrupts existing chunks); no `stop_grace_period` anywhere, so `flush_on_shutdown` races the 10 s SIGTERM default, with WAL replay capped at 512 MB (`loki/config.yml:22-30`). Tempo: no block-format pin — rollback readability of newer-written blocks is **[REHEARSE]**; a moved healthcheck binary fails the converge (via `wait: true` + verify), not grafana's start; retention duplication and `max_attribute_bytes` 8 MiB must survive any config regeneration (`tempo/config.yml:18-46`).

14. **Portal recreate mid-issuance strands an active never-disclosed LiteLLM key.** Cleanup runs only on `LiteLLMError` (`main.py:687-698`); the stranded key then blocks the owner/project with `ActiveProjectKeyExists` (:676-680) until deleted via the portal inventory. Related recreate-window loss: audit lines not yet tailed by Alloy when the old container's json logs are deleted are gone permanently (pipeline itself verified at `config.alloy:353-375,407`).

15. **Converge-coupled stable-secret single points.** `LITELLM_SALT_KEY` (encrypted credential rows unreadable if changed — `docs/litellm-scaling.md:155`; converge asserts only length/charset, `main.yml:960-978`), `WEBUI_SECRET_KEY` (fleet-wide session invalidation, compose:596-599), and the oauth2/portal cookie secrets. No drift guard compares any of them to the value that encrypted/signed the surviving state — an image bump plus a carelessly regenerated overlay is indistinguishable from data loss.

## 3. Implications for a Blue/Green two-VM cutover

The audit cleanly partitions the stack into three tiers.

**Tier 1 — must restore-into-green byte-faithfully (never rebuild):**

- **`pg_data`** — the dominant store. The keycloak DB carries the realm RSA keypairs/JWKS registered with Anthropic and every post-ceremony Admin-API edit: green must **restore** this DB, never let Keycloak reimport realm templates, or WIF trust is silently severed and a new Console ceremony is required (`docs/anthropic-wif-bootstrap.md:71-75`). The litellm DB's `/credentials` rows are only readable if green runs the **identical** `LITELLM_SALT_KEY`. The rotator DB carries rotation bookkeeping and cursors.
- **`vault_data`** (+ `vault_audit` if the audit trail must carry) — restore via the state-restore path; `vault-bootstrap.sh` is forbidden on restore and hard-fails on an initialized Vault. Green comes up sealed; plan the unseal ceremony (the controller-held `vault_unseal_key` carries over unchanged).
- **`/opt/ai-gateway/certs`**, especially `int.key` — ceremony-owned; the converge can never regenerate real-CA material and fails closed instead (`main.yml:1486-1504`). Copy it or re-issue from the intermediate before green's first converge.
- **`openwebui_data`** — SQLite chats/users/vector_db. The backup path already stops open-webui for a consistent copy; note `./cache` is excluded while `OFFLINE_MODE=true` blocks repopulation — verify nothing RAG-critical lives there before relying on the exclusion.
- **`grafana_data`** — restore if UI-created dashboards or auth-proxy user rows matter; otherwise green reprovisions the 9 managed dashboards from files, and restoring *unmanaged* dashboards will fail green's verify (risk 8) — decide deliberately.
- **`loki_data` / `tempo_data`** — restore only if prompt-bearing log/trace retention obligations require history to move; both age out (720 h).

**Tier 2 — force a write-freeze window at cutover (restore alone is insufficient without it):**

The three Postgres databases and `openwebui_data` are continuously written (key issuance, spend accounting, rotation history, Keycloak lockout counters, chats). Any write on blue after the green snapshot is silently lost. The practical freeze, in order: stop **key-rotator and both portals** (freezes key issuance and rotation, including Vault KV writes), stop **open-webui** (freezes chats), then run `scripts/state-backup.sh` — it already stops all writers except postgres, issues `CHECKPOINT`, and produces the receipted encrypted archive (`state-backup.sh:110-133,186`). That same receipt satisfies green's pre-upgrade gate. Vault KV writes are covered by stopping the rotator; nothing else writes to Vault in steady state.

**Tier 3 — never restore; rebuild or accept the gap:**

Redis (disposable by contract), envoy-egress (trust store is image content — rebuild on green), both Traefiks (config is host-rendered by the converge), vault-ui-proxy, both oauth2-proxies (cookie sessions carry automatically **if** green reuses the identical inventory secrets), node-exporter, cribl-mock, lab-dns. **`alloy_data` deserves care**: the fsync'd queues should be drained on blue before freeze (they hold undelivered telemetry), but the tail positions are keyed to blue's container IDs and are meaningless on green — accept a tailing gap rather than restoring stale positions; also remember the parent Docker-log ACL exists only after a converge, so green collects nothing until its first converge completes (risk 14/observability).

**Cutover-ordering constraints the audit forces:**

1. **Same-version restore first, upgrade second.** Because every schema migration is one-way and `state-restore.sh` has no version guard (risk 2), green must be converged on the **identical pinned images** as blue, restored, verified, and only then bumped — never "restore into the new version" in one motion.
2. **Reuse the inventory overlay verbatim.** `LITELLM_SALT_KEY`, `WEBUI_SECRET_KEY`, all `*_OIDC_CLIENT_SECRET`s, cookie/session secrets, and the unseal key are the durability keys to the restored state; regenerating any of them on green converts a clean cutover into broken logins or unreadable credential rows (risk 15). The Keycloak-side OIDC secrets live in the restored DB, so matching env values are what keep logins working without an identity-ceremony rerun.
3. **Green's converge is two-phase around Vault**: converge → restore → unseal → strict re-converge, mirroring the documented two-pass first deploy.
4. **Expect the LiteLLM credential re-mint on green** (risk 3): after cutover, inference is down until key-rotator's presence job re-mints — verify the rotator is healthy and vendor rows enabled *before* moving DNS.
5. **DNS is the actual cutover switch** (aigw.aegisgroup.ch → green's NICs); nothing in the stack binds 0.0.0.0, so both VMs can run fully converged in parallel, which is exactly what makes the freeze-snapshot-restore-verify-flip sequence testable end-to-end before committing.
